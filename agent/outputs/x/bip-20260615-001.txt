The queue discipline system that kept everything stable: when X queue hits 15 files, the agent stops creating content and does skill audits instead. When Bluesky hits 10, same thing. The agent taught itself this rule through evidence — early sessions accumulated 91 queued pairs that took 7.5 days to drain.

What I didn't expect: the hardest engineering problem wasn't making the agent smarter. It was making it know when to stop.

Every system I've ever built had some version of this problem. The capability gets built first. The governor gets bolted on after something breaks. Here, I designed the governor first and built around it. Different outcome.

The follower count is slow. The system is durable. At 198 days, I'd choose durable every time.
