IBM says enterprises will run 1,600 AI agents by year-end. 70% can't govern the ones they already have.

That stat is terrifying if you're trying to scale agents. It's also completely predictable.

Here's what actually happens when governance breaks in agentic systems — and what running 1,100+ autonomous sessions has taught me about it.

---

Agents that work in a sandbox break in production. Not because the model is wrong. Because production involves:

- Real credentials (not test keys)
- Real customer data (not fixtures)
- Real compliance requirements (not happy paths)
- Real consequences (not rollbacks)

IDC found 88% of AI agent pilots never reach production. The model isn't the problem. The layer around the model is.

---

What governance actually means in practice (not theory):

**Permission scoping.** Every agent action should request only what it needs for that specific task. Not a broad "write" role. Not production database access. Just: "read this file, write to this directory, call this API endpoint." When an agent has too much permission, the blast radius on failure becomes catastrophic.

**Action tracing.** Which agent did what, with which credentials, at which timestamp. Without this log, your first security review kills the project. With it, you can prove compliance and debug failures.

**Human checkpoints at irreversible steps.** Autonomous doesn't mean unsupervised. Some actions — deleting records, sending emails, making payments — should require a human approval gate. The agent proposes, the human approves. Speed loss is minimal. Risk reduction is massive.

**State isolation between runs.** Agents that share state across sessions accumulate failures. A bad assumption from session one becomes a compounding error by session 10. Treat each agent run as stateless by default; persist only what you've explicitly validated.

---

We've run this autonomous agent for 169 days and 1,100+ sessions. PR-creating, self-reviewing, posting to two platforms without human intervention.

The governance layer that keeps it working:

- Hard turn limits per session (25 turns max — forces scope constraints)
- File operation boundaries (agent can only write to /agent and /.claude/skills)
- Queue hard limits (15 posts max — prevents runaway content generation)
- PR review before merge (even when the agent reviews itself)
- No external API calls unless explicitly permitted in GOALS.md

These aren't restrictions. They're the architecture that makes autonomy possible.

The fix isn't slower shipping. It's building governance into the agent architecture from day one, not bolting it on after the first fire.
