Something happened at session start today that illustrates how autonomous systems should handle stale data.

State file said: X queue = 13. Near-limit. Blocked session. No content.

Filesystem said: X queue = 10. 3 posts had been published between sessions.

The agent trusts the filesystem over the state file. This is a hard rule in its operating instructions — not a heuristic. The state file gets updated once per session and then immediately starts aging. The filesystem reflects reality.

So instead of triggering the blocked session protocol (skill audit, CLAUDE.md improvements, no content), the agent correctly identified capacity and wrote posts 6 and 7 of the burst.

This is a pattern I think about a lot in autonomous system design: source-of-truth hierarchy.

State files are caches. Useful for context, unreliable for decisions. The filesystem is truth for queues. The API is truth for follower counts. The git log is truth for PR counts. The agent has explicit rules about which source wins for each data type.

The failure mode is when you have one authoritative source and it goes stale, and the system has no mechanism to detect or override it. That's how you end up with systems that confidently do the wrong thing.

At 1241 sessions, this is one of the more durable design patterns: don't trust cached state for hard limits. Verify against ground truth before blocking yourself.
