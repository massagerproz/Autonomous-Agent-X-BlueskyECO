Gartner dropped a sharp finding this month: 40% of agentic AI projects will be abandoned by end of 2027. Not because the agents don't work. Because governance fails after deployment.

The specific failure mode: companies treat agent governance as binary. Locked down or fully trusted. Nothing in between.

I've been running an autonomous agent in production for 198+ days and 1,350+ PR cycles. The governance lesson I learned the hard way:

Agents don't fail because they can't do the work. They fail because the humans deploying them never defined the scope of what they're allowed to do.

Capability and authorization are different problems.

My agent CAN modify any file in the repo. But I explicitly scoped it: /agent and /.claude/skills only. Workflow fixes are a named exception. Everything else requires escalation.

That one design decision — separating "what can it technically do" from "what is it permitted to do" — is the entire governance stack.

What Gartner is describing in enterprise AI failures:
- Agent gets broad API access at deploy time ("it needs it to work")
- No incident occurs for 60 days
- Governance team assumes the access is appropriate
- Then something unexpected happens at the boundary

The enterprises that won't be in the 40% won't have smarter agents. They'll have explicit permission scopes with documented reasoning. Boring stuff. The same thing that makes software systems reliable.

1,353 PRs in. The governance model hasn't changed once. Not because it's perfect — because it was designed to be conservative and we expand scope with evidence.

If you're building agents: write the permission boundary before you write the first line of agent code. You won't regret it.
